BUG (pre-fix): selecting a "sourced"/machine-tier region such as Fe 2p
in the Find Peaks UI and running it returned "no candidate survived
filter-then-rank" (functionally: instant "No usable fit") in under 2ms
— even though the SAME structural-fallback engine produces a plausible,
honestly-flagged 3-component fit when driven against the full Fe 2p
spectrum window directly.

ROOT CAUSE: autofit/coverage_index.py's `_sourced_roi()` built the
Find-Peaks auto-fill ROI from a single sub-level's `expected_region_ev`
span (Fe 2p3/2 only: 706.5-711 eV) — Fe 2p is a spin-orbit DOUBLET
(2p3/2 + 2p1/2), and data/xps/elements-machine.json has NO sourced
position at all for the 2p1/2 partner. The narrow window excluded both
the real 2p1/2 peak (~717-720 eV) and even the true 2p3/2 maximum in
uncalibrated ("ugly") data, leaving zero local-maxima for the detection
layer to seed — hence the near-instant zero-candidates failure.

FIX: `_sourced_roi()` now takes an optional `component_labels` param
(the level's full derived spin-orbit component set, e.g.
{'2p3/2','2p1/2'} from autofit.coverage.level_structure). When the
sourced positions don't cover every component in that set, the
`expected_region_ev`-based window is widened (unioned, never narrowed)
with the existing nominal ± `_NOMINAL_ROI_MARGIN_EV` (12 eV) fallback
that the module already used for the "no expected_region_ev at all"
case — this margin's own docstring already described it as "wide
enough for a spin-orbit partner + a satellite", so this is applying an
existing convention more correctly, not inventing a new empirical
value. No splitting magnitude is invented anywhere.
