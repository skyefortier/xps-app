BACKGROUND -- what this unit is supposed to be:

`autofit/candidates.py`'s CWT ridge detector had a fixed scale ladder:
FWHM 0.3-2.4 eV, 8 geometric steps. The 2.4 eV ceiling existed only because
it was "just above FWHM_MAX_ORDINARY_EV = 2.0" -- a chemistry-derived
"ordinary peak width" constant leaking into the signal-processing layer,
capping what the detector could physically characterize. This commit
decouples exactly TWO of the three roles that constant played (Skye's
explicit split, by MEANING not "it's the same constant"):

  (i)   the detector's own scale ceiling (CWT_FWHM_MAX_EV, retired) --
        CHARACTERIZATION: how well the detector can measure what's there.
  (ii)  the curvature seed's fwhm_init clip in build_candidate_pool --
        also CHARACTERIZATION (the optimizer's starting estimate).
  (iii) the fit's free-parameter bound on preseed_curvature_* slots in
        _preseed_augmented (engine.py) -- a DIFFERENT question, what a
        component may BECOME (degeneracy control). Deliberately OUT OF
        SCOPE for this unit, deferred to migration step 6. MUST be
        UNTOUCHED in this diff.

Both (i) and (ii) now derive from the ROI's own point count and grid step,
via the SAME kernel-fits-in-window bound (`radius = ceil(4*sigma) <= (n-1)/2`)
the per-scale filter already enforced elsewhere in this module -- the
stated intent is NO new magic number, just a different anchor for the
existing one. Scale-ladder density (_cwt_n_scales) is derived to preserve
the ORIGINAL calibration's steps-per-octave rather than staying fixed at 8
regardless of range.
