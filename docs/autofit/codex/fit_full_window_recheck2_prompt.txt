ROUND-2 RECHECK'S TWO FINDINGS (verbatim, both from the same NO-GO run):

FINDING 1: "`match_components_to_slots()` now uses the widened
effective window center for disambiguation. That can relabel a
component into a neighboring slot even when it is still inside its
original literature window. Example: C 1s graphitic (284.0, 284.8)
widened low to ROI 270..300, adjacent aliphatic (284.6, 285.4), fitted
component at 284.65. It is accepted by both, but graphitic's widened
center becomes 277.4, so [the disambiguation] chooses aliphatic. This
can tank graphitic persistence in stability. Acceptance needs widened
bounds; identity tie-breaking should stay anchored to the original slot
window."

FINDING 2: "proposal blocking now uses full-window overrides for
populated `region == "unassigned"` slots. Since
`_full_window_bound_overrides()` maps unassigned primary slots to the
full ROI, any accepted proposal/preseed/detection slot blocks every
later residual proposal in the ROI when `fit_full_window=True`. That
breaks the documented iterative proposal behavior
(`PROPOSAL_MAX_PER_CANDIDATE = 3`) under this option."

FIXES APPLIED (same session, commit a9e2b0e — please verify against
current disk state):

1. (Finding 1) `_window_center()` inside `match_components_to_slots()`
   (autofit/engine.py) no longer accepts/uses a `bound_override` at
   all — it always calls `_effective_be_window(slot, components)` with
   NO override, so the tie-break reference point stays anchored to the
   slot's TRUE window regardless of `fit_full_window`. Only `_accepts()`
   (the actual accept/reject test, a few lines above) still uses the
   override. New test:
   `test_full_window_tie_break_never_uses_the_widened_center` in
   tests/autofit/test_fit_full_window_option.py — reproduces the exact
   graphitic/aliphatic/284.65 scenario from the finding, confirmed to
   FAIL pre-fix (resolved to aliphatic / `slot_map["graphitic"] is
   None`) and pass post-fix.

2. (Finding 2) `_proposal_blocked()` and `_detect_residual_proposals()`
   had their `bound_overrides`/`fit_full_window` parameters REMOVED
   entirely — fully reverted to their pre-this-feature signatures and
   behavior (this was a speculative addition never required by the
   original round-1 review; round-1's actual finding was only about
   `run_stability_analysis`/`match_components_to_slots`). Their call
   sites inside `compare_models()` no longer pass `fit_full_window` or
   `bound_overrides` either. New test:
   `test_proposal_blocking_never_widens_a_populated_slots_territory`.
