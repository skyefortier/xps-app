RUN 1's FINDING (verbatim): "widened curated fits are not accepted by
the stability/identity layer. The optimizer bounds widen, but
run_stability_analysis() still classifies refit components against the
original slot.be_window. ... a highest-BE curated primary slot can fit
just above its original literature window, exactly as intended. But in
stability, that same fitted component is outside _effective_be_window(),
becomes an orphan, the slot's persistence stays low/zero, and the
candidate is filtered or marked implausible."

RUN 2's FINDING (verbatim): "_full_window_bound_overrides() can tighten
or even invert curated outer-slot bounds when fit_full_window=True and
the selected ROI is not wider than the literature envelope. ... a C 1s
candidate with a lowest slot window (284.0, 285.0) and ROI 287.0..300.0
gets center bounds (287.0, 285.0) for that slot."

FIXES APPLIED (same session, please verify against current disk state —
do not trust this description alone):

1. (Run 2's bug) `_full_window_bound_overrides()` in autofit/engine.py:
   the touched side of a curated outer slot's bound now uses
   `min(slot.be_window[0], roi_lo)` (low side) / `max(slot.be_window[1],
   roi_hi)` (high side) against the slot's OWN original bound, instead
   of assigning a bare ROI edge — so it can only ever widen, never
   narrow or invert. New test:
   `test_full_window_never_narrows_or_inverts_a_bound_when_roi_is_shifted`
   in tests/autofit/test_fit_full_window_option.py.

2. (Run 1's bug) `_effective_be_window()` now accepts an optional
   `bound_override` parameter; `match_components_to_slots()` now accepts
   a `bound_overrides` dict (role -> (lo, hi)) and passes the right one
   to each `_effective_be_window()` call. `run_stability_analysis()`
   computes `_full_window_bound_overrides(model, x)` once (when
   `fit_full_window`) and passes it through to
   `match_components_to_slots()`. The same `bound_overrides` threading
   was ALSO applied to `_proposal_blocked()` and
   `_detect_residual_proposals()` (a related, lower-severity instance of
   the same class — the proposal-eligibility "populated slot's window"
   check) for consistency, though this second path wasn't explicitly
   named in run 1's finding. New test:
   `test_full_window_stability_does_not_orphan_a_widened_component` —
   confirmed to FAIL against the pre-fix code (reverted just the
   `match_components_to_slots(...)` call's `bound_overrides` argument
   locally, re-ran, watched it fail with `persistence=0.0` — matching
   run 1's predicted failure mode exactly — then restored the fix and
   watched it pass).
