ROOT CAUSE (found by direct tracing): `updatePlot()`
(templates/index.html) freezes the chart's background/fit-curve
rendering to `state.fitResult`'s own `be`/`bgIntensity` arrays whenever
`state.fitResult` is set with matching-length `be`/`bgIntensity` (the
"haveFit" branch) — this happens after ANY prior manual Run Fit, or
Auto-Fit C1s Graphite. `applyFindPeaks()` (the function that applies
Find-Peaks-suggested peaks to the active tab) NEVER touched
`state.fitResult` at all, so if a PRIOR fit had frozen the chart to a
narrower range, applying NEW Find-Peaks peaks left the chart still
frozen to that OLD, narrower range — completely unrelated to how wide a
window Find Peaks itself used internally. This is a pure FRONTEND
rendering-staleness bug; the backend's x/y array is always exactly the
full requested ROI (verified in an earlier session), and Find Peaks'
own JSON response carries no `be`/`fittedY`/`bgIntensity` arrays to
rebuild a proper `state.fitResult` from at all.

THE FIX (templates/index.html only, zero backend/engine changes):
1. Where `_fpLast` gets set after a completed analysis (search for
   `_fpLast = { body, method, regions`), it now also records
   `fitFullWindow: !!options.fit_full_window` — the flag that was
   actually used for THIS specific analysis run.
2. In `applyFindPeaks()`, right before `renderPeakList(); updatePlot();`,
   a new line: `if (_fpLast && _fpLast.fitFullWindow) state.fitResult =
   null;` — ONLY when the completed run used the flag. This makes
   `updatePlot()`'s "haveFit" check false, so it falls back to its
   EXISTING (already well-tested, unrelated to this fix) unfit-preview
   rendering path: `getROIData()` (reads the LIVE `#roi-min`/`#roi-max`
   DOM fields — the SAME fields `runFindPeaks()` itself read when
   building the request) + client-side `computeBackground()`. Default
   (unchecked) path is completely untouched — `state.fitResult` is left
   exactly as it was, byte-for-byte the same behavior as before this
   commit.
