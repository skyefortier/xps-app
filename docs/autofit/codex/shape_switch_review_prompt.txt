THE BUG: in templates/index.html, cycling a peak's "Line shape" dropdown
(DS+G -> another shape -> back to DS+G) silently CHANGED its parameters with
NO Run Fit — observed DS+G laM 3.11 -> 0.40 and laBeta 0.05 -> 0.30 on a
round-trip. Cause: the old _switchPeakShape deleted the old shape's params
(_clearShapeSpecificParams) then re-seeded the new shape's DEFAULTS
(_applyShapeDefaults), so a round-trip reset every fitted shape param.

THE FIX (all in templates/index.html, around the SHAPE_PARAM_SCHEMA /
_switchPeakShape region — grep _applyShapeSwitch / _widthField): a new pure
_applyShapeSwitch(peak, newShape) that (1) carries the effective WIDTH across
the switch — DS+G stores width in laM (Gauss FWHM), every other shape in the
top-level fwhm; the eV width + its lock is mapped across the laM<->fwhm
boundary — and (2) preserves every carried-over param (never deletes),
seeding a default ONLY for a param the new shape genuinely introduces that
the peak lacks. _switchPeakShape now calls it per family member.

RAILS (verify): branch-only; touches ONLY the shape-switch behavior; NO
change to fitting math, the optimizer (runFit), evalPeak/evalPeakArray, or
/api/fit / the backend. Confirm the diff (git show d5c3222) is confined to
the schema/_switchPeakShape region + the new JS test.
