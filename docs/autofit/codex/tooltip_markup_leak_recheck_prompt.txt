ROUND 1 FINDING (both of 2 independent runs, unanimous NO-GO on test
realism only — both runs confirmed the PRODUCTION fix itself, the
statusText/statusHtml split in `_fpRenderResults()`, was correct and
found no other leak in the Find Peaks section): the new test's
synthetic payload modeled an unrealistic backend shape. It named the
winning candidate `"A0_graphite_only+bfix"` (a `decisive_override`
bound-fixed refit) and gave it a truthy `filter_reason`, but
`autofit/engine.py`'s `_apply_decisive_override()` renames the promoted
refit's report to `<original>+bfix` and appends it directly to
`result.survivors` — it is NEVER also appended to `result.filtered_out`.
`build_analysis_record()` (autofit/methods/ic_model_comparison.py)
looks up `filter_reason` by exact model name from `filtered_out`, so a
real `+bfix` winner always gets `filter_reason: None`. Both runs
identified the REAL trigger instead: `rank_and_filter()`'s
`no_clean_survivor` conditional tier (autofit/engine.py) promotes the
SAME `ModelReport` objects that were just appended to `filtered_out`
(tagged with their plausibility violation) directly into `survivors`
when nothing passed cleanly but they're otherwise stable — same
object, same `model.name`, no suffix — so the winner's own name is
simultaneously a key in `filtered_reason` (truthy) and in
`survivor_rank` (`survived: True`).
